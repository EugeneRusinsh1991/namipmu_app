# Анализ визуального дублирования и предложение по оптимизации

## 🔍 Текущая проблема
На основе анализа `src/styles/design-system/theme.ts` и `DESIGN_SYSTEM_GUIDE.md`, тип `SemanticTokens` включает в себя как цветовые значения, так и геометрические. 

Когда мы создаем `lightTheme` и `darkTheme`, мы вынуждены полностью описывать объект `SemanticTokens`. Это приводит к тому, что в обоих объектах дублируются:
1. **Borders**: `radiusSm`, `radiusMd`, `radiusLg`, `radiusFull`, `widthThin` и т.д.
2. **Spacing**: Вся шкала отступов (`xs`, `sm`, `md`, `lg`, `xl`, `xxl`).
3. **Typography**: Размеры шрифтов (`fontSizeXs-Xxl`), начертания (`fontWeight`) и межстрочные интервалы.
4. **Shadows (Геометрия)**: Смещение (`offset`), радиус размытия (`blur`) и распространение (`spread`).

## 🚩 Где найден дублирующийся код

### 1. `src/styles/design-system/theme.ts`
Сейчас это "монолитный" конфиг темы. Если завтра ты захочешь изменить `radiusMd` с 12 на 16, тебе придется менять это в двух местах: и в `lightTheme`, и в `darkTheme`. Если появится третья тема (например, "High Contrast"), правок станет еще больше.

### 2. `src/styles/design-system/components.ts`
Функция `getComponentSpecs(tokens)` принимает в себя все токены. Поскольку геометрия (высота кнопок, отступы в карточках) "зашита" в `tokens`, она пересчитывается при каждой смене темы, хотя значения остаются идентичными.

### 3. Скрытый конфликт с `src/styles/content-dimensions.ts`
У тебя есть отдельный файл `content-dimensions.ts`, где также описаны `SPACING` и `IMAGE_SIZES`. Это создает "второй источник правды". Например, отступ `md` может быть определен и в теме, и в этом файле.

## 💡 Идея оптимизации: Разделение на "Foundation" и "Theme"

Чтобы достичь твоей цели (менять радиус в одном месте), нужно разделить токены на две категории:

### 1. Visual Foundation (Константы)
Файл, который не знает о существовании тем. В нем живут:
- **Геометрия**: Радиусы, толщины границ.
- **Лейаут**: Шкала отступов (spacing).
- **Типографика**: Шкала размеров и семейства шрифтов.
- **Спеки теней**: Настройки теней без учета цвета (или с использованием прозрачного черного).

### 2. Color Theme (Переменные)
Объекты `lightTheme` и `darkTheme` должны содержать **только цвета**:
- `surface`
- `text`
- `interactive` (цвета состояний)
- `shadowColor` (единственный параметр тени, который меняется)

## 🛠 Как это будет выглядеть (План рефакторинга)

1. **Создать `foundation.ts`**:
   Перенести туда `borders`, `spacing`, `typography` из `theme.ts`.

2. **Очистить `theme.ts`**:
   Оставить в `SemanticTokens` только цветовые категории.

3. **Обновить `useDesignTokens.ts`**:
   Хук должен собирать финальный объект из двух частей:
   ```typescript
   const tokens = { 
     ...foundation, 
     colors: currentThemeColors 
   };
   ```

4. **Синхронизировать `content-dimensions.ts`**:
   Сделать так, чтобы этот файл импортировал значения из `foundation.ts`, а не хардкодил их заново.

## ✅ Результат
- **Единая точка правды**: Радиус скругления кнопок меняется в одном файле `foundation.ts`.
- **Чистота тем**: Файл `theme.ts` становится в 2 раза меньше и содержит только цветовую логику.
- **Производительность**: React Native не нужно пересчитывать геометрические свойства при переключении темы.

## 🧩 Унификация фонов контентных блоков (Line 100+)

### 🔍 Проблема "Разрозненных Поверхностей"
На текущий момент `Card`, `Quiz`, `Checklist` и `Timer` являются логически однотипными "блоками контента". Однако, если в спецификациях (`components.ts`) они привязаны к разным токенам (например, `surfacePrimary` для карточек и `surfaceSecondary` для квизов), это создает визуальный шум. Изменение фона "всех блоков" в одной теме сейчас требует правок в нескольких местах.

### 💡 Решение: Семантическая роль "Container Surface"
Чтобы элементы не выглядели вырванными из контекста, необходимо внедрить единый токен для всех блочных контейнеров.

#### 1. Расширение `SemanticTokens` (в `theme.ts`)
В категории `surface` мы выделяем (или закрепляем существующий) токен `surfacePrimary` (или создаем `surfaceContainer`) как основной фон для всех интерактивных блоков контента.

#### 2. Маппинг в `components.ts`
Спецификации всех компонентов-контейнеров должны ссылаться на этот единый токен:
```typescript
specs.card.large.backgroundColor = tokens.surface.surfacePrimary;
specs.quiz.container.backgroundColor = tokens.surface.surfacePrimary;
specs.checklist.container.backgroundColor = tokens.surface.surfacePrimary;
specs.timer.container.backgroundColor = tokens.surface.surfacePrimary;
```

### ✅ Результат
- **Визуальная целостность**: Все блоки в приложении выглядят как части единой системы, имея одинаковый фон и "глубину".
- **Мгновенное обновление**: Если в `darkTheme` ты решишь сделать фон всех блоков чуть светлее (например, `neutral800` вместо `neutral850`), тебе достаточно изменить **одно** значение `surfacePrimary`.
- **Устранение дублирования**: Логика "какой фон у контента" теперь живет только в теме, а компоненты просто потребляют этот стандарт.
